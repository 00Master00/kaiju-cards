# บทที่ 1: บทนำ

## 1.1 ความเป็นมาและความสำคัญของโครงการ

ในยุคดิจิทัลที่เทคโนโลยีสารสนเทศมีบทบาทสำคัญต่อการดำเนินธุรกิจและการให้บริการ การพัฒนาเว็บแอปพลิเคชันจึงเป็นหนึ่งในกระบวนการสำคัญที่ต้องมีการควบคุมคุณภาพอย่างเข้มงวด การทดสอบซอฟต์แวร์ (Software Testing) จึงเป็นกระบวนการที่สำคัญในการรับรองคุณภาพของซอฟต์แวร์ก่อนนำไปใช้งานจริง

ระบบจัดการข้อมูลอนิเมะ (Anime Management System) เป็นเว็บแอปพลิเคชันที่ถูกพัฒนาขึ้นเพื่อให้บริการจัดเก็บและจัดการข้อมูลอนิเมะ โดยมีทั้งส่วนของผู้ดูแลระบบ (Back Office) และส่วนของผู้ใช้งานทั่วไป (Front Office) การทดสอบระบบดังกล่าวจึงต้องครอบคลุมทั้งการทำงานของระบบในระดับต่างๆ ตั้งแต่การทดสอบหน่วยย่อย (Unit Testing) ไปจนถึงการทดสอบระบบโดยรวม (System Testing)

โครงการสหกิจศึกษานี้จึงมุ่งเน้นไปที่การออกแบบ Test Case และการวิเคราะห์ผลการทดสอบสำหรับเว็บแอปพลิเคชันดังกล่าว โดยใช้เทคนิคการทดสอบทั้ง White Box Testing และ Black Box Testing เพื่อให้ครอบคลุมการทดสอบทุกมิติของระบบ

### ปัญหาที่พบในการพัฒนาซอฟต์แวร์

1. **ขาดการทดสอบที่เป็นระบบ** - หลายโครงการพัฒนาซอฟต์แวร์ไม่มีการวางแผนการทดสอบที่ชัดเจน ทำให้พบข้อผิดพลาดเมื่อนำไปใช้งานจริง
2. **Test Cases ไม่ครอบคลุม** - การออกแบบ Test Cases ที่ไม่ครอบคลุมทำให้พลาดการตรวจสอบกรณีการใช้งานบางส่วน
3. **ขาดการวิเคราะห์ผลการทดสอบ** - การทดสอบที่ไม่มีการวิเคราะห์ผลอย่างเป็นระบบ ทำให้ไม่สามารถปรับปรุงคุณภาพของระบบได้อย่างมีประสิทธิภาพ
4. **ขาดการทดสอบอัตโนมัติ** - การพึ่งพาการทดสอบด้วยมือเพียงอย่างเดียวทำให้เสียเวลาและมีความเสี่ยงในการพลาดข้อผิดพลาด

### ความสำคัญของโครงการ

โครงการนี้มีความสำคัญในการ:
- พัฒนาทักษะการออกแบบ Test Cases ที่มีคุณภาพและครอบคลุม
- ฝึกฝนการใช้เครื่องมือทดสอบอัตโนมัติในการพัฒนาเว็บแอปพลิเคชัน
- เรียนรู้การวิเคราะห์ผลการทดสอบและการจัดทำรายงาน
- สร้างความเชื่อมั่นในคุณภาพของระบบก่อนนำไปใช้งานจริง

## 1.2 วัตถุประสงค์ของโครงการ

### วัตถุประสงค์หลัก

1. **เพื่อออกแบบ Test Cases ที่ครอบคลุม**
   - ออกแบบ Test Cases สำหรับ White Box Testing (Automated Testing)
   - ออกแบบ Test Cases สำหรับ Black Box Testing (Manual Testing)
   - ครอบคลุมการทดสอบทุกฟังก์ชันหลักของระบบ

2. **เพื่อพัฒนาการทดสอบอัตโนมัติ (Automated Testing)**
   - พัฒนา Unit Tests สำหรับการทดสอบหน่วยย่อยของโค้ด
   - พัฒนา Integration Tests สำหรับการทดสอบการทำงานร่วมกันของส่วนประกอบ
   - พัฒนา Component Tests สำหรับการทดสอบคอมโพเนนต์ React

3. **เพื่อดำเนินการทดสอบด้วยตนเอง (Manual Testing)**
   - ทดสอบ User Interface และ User Experience
   - ทดสอบฟังก์ชันการทำงานของระบบตามสถานการณ์จริง
   - ทดสอบ Cross-browser Compatibility

4. **เพื่อวิเคราะห์และจัดทำรายงานผลการทดสอบ**
   - วิเคราะห์ผลการทดสอบอัตโนมัติ (Test Results, Code Coverage)
   - วิเคราะห์ผลการทดสอบด้วยตนเอง (Defects, Issues)
   - จัดทำรายงานสรุปผลการทดสอบและข้อเสนอแนะ

5. **เพื่อพัฒนาทักษะการทดสอบซอฟต์แวร์**
   - ฝึกฝนการใช้เครื่องมือทดสอบ (Jest, React Testing Library)
   - เรียนรู้เทคนิคการออกแบบ Test Cases
   - พัฒนาความเข้าใจในกระบวนการทดสอบซอฟต์แวร์

### วัตถุประสงค์เฉพาะ

6. **เพื่อประเมินคุณภาพของระบบ**
   - วัดผลการทดสอบเชิงปริมาณ (Test Pass Rate, Coverage %)
   - วัดผลการทดสอบเชิงคุณภาพ (Defect Density, Severity)

7. **เพื่อสร้างมาตรฐานการทดสอบ**
   - สร้างเอกสาร Test Plan และ Test Strategy
   - สร้างเทมเพลตสำหรับการออกแบบ Test Cases
   - สร้างแนวทางในการรายงานข้อบกพร่อง (Bug Report)

8. **เพื่อปรับปรุงคุณภาพของระบบ**
   - ระบุข้อบกพร่องและจุดที่ต้องปรับปรุง
   - เสนอแนะแนวทางในการแก้ไขปัญหา
   - ติดตามและตรวจสอบการแก้ไขปัญหา

## 1.3 ขอบเขตของโครงการ

### 1.3.1 ระบบที่ทำการทดสอบ

**Anime Management System** - ระบบจัดการข้อมูลอนิเมะที่ประกอบด้วย:

#### Back Office (ส่วนผู้ดูแลระบบ)
- **จัดการข้อมูลอนิเมะ** (Anime Management)
  - เพิ่ม/แก้ไข/ลบข้อมูลอนิเมะ
  - อัพโหลดรูปภาพและข้อมูลต่างๆ
  - จัดการสถานะการแสดงผล
  
- **จัดการหมวดหมู่** (Genre Management)
  - เพิ่ม/แก้ไข/ลบหมวดหมู่
  - กำหนดสีและไอคอนของหมวดหมู่
  
- **ระบบตรวจสอบสิทธิ์** (Authentication)
  - เข้าสู่ระบบ/ออกจากระบบ
  - การจัดการสิทธิ์การเข้าถึง

#### Front Office (ส่วนผู้ใช้งานทั่วไป)
- **หน้าแรก** (Home Page)
  - แสดงอนิเมะแนะนำ
  - แสดงอนิเมะยอดนิยม
  
- **หน้าค้นหา** (Search Page)
  - ค้นหาอนิเมะตามชื่อ
  - กรองตามหมวดหมู่
  - เรียงลำดับผลการค้นหา
  
- **หน้ารายละเอียด** (Detail Page)
  - แสดงข้อมูลอนิเมะอย่างละเอียด
  - แสดงหมวดหมู่ที่เกี่ยวข้อง

### 1.3.2 ประเภทการทดสอบที่ดำเนินการ

#### White Box Testing (Automated Testing)
1. **Unit Testing**
   - ทดสอบ Custom Hooks (useAnimeData, useGenres)
   - ทดสอบ Utility Functions
   - ทดสอบ Context Providers

2. **Component Testing**
   - ทดสอบ React Components แบบแยกส่วน
   - ทดสอบการแสดงผลและ Props
   - ทดสอบ Event Handlers

3. **Integration Testing**
   - ทดสอบการทำงานร่วมกันของคอมโพเนนต์
   - ทดสอบ Data Flow ระหว่างส่วนประกอบ
   - ทดสอบการเชื่อมต่อกับ API

#### Black Box Testing (Manual Testing)
1. **Functional Testing**
   - ทดสอบฟังก์ชันการทำงานทุกส่วน
   - ทดสอบตาม User Stories
   - ทดสอบ Business Logic

2. **User Interface Testing**
   - ทดสอบการแสดงผลบนหน้าจอ
   - ทดสอบ Responsive Design
   - ทดสอบ Cross-browser Compatibility

3. **Usability Testing**
   - ทดสอบความสะดวกในการใช้งาน
   - ทดสอบ User Experience
   - ทดสอบ Navigation Flow

4. **Performance Testing**
   - ทดสอบความเร็วในการโหลดหน้า
   - ทดสอบการตอบสนองของระบบ
   - วัด Performance Metrics

### 1.3.3 เครื่องมือและเทคโนโลยีที่ใช้

#### เครื่องมือทดสอบอัตโนมัติ
- **Jest** - JavaScript Testing Framework
- **React Testing Library** - React Component Testing
- **MSW (Mock Service Worker)** - API Mocking

#### เทคโนโลยีพัฒนา
- **React 18** - Frontend Framework
- **TypeScript** - Type-safe JavaScript
- **Vite** - Build Tool
- **Tailwind CSS** - CSS Framework
- **Supabase** - Backend as a Service

#### เครื่องมือช่วยเหลืออื่นๆ
- **Chrome DevTools** - Debugging และ Performance Analysis
- **Git/GitHub** - Version Control
- **VS Code** - Code Editor

### 1.3.4 ขอบเขตที่ไม่รวมในโครงการ

โครงการนี้ **ไม่ครอบคลุม** ถึง:
1. การทดสอบด้านความปลอดภัย (Security Testing) เชิงลึก
2. การทดสอบภาระ (Load Testing) ในระดับสูง
3. การทดสอบบน Mobile Applications
4. การทดสอบ Accessibility (A11y) อย่างละเอียด
5. การทดสอบ API Performance ระดับ Enterprise

## 1.4 ประโยชน์ที่คาดว่าจะได้รับ

### 1.4.1 ประโยชน์ต่อผู้ปฏิบัติงาน (นักศึกษาสหกิจ)

1. **ความรู้และทักษะด้านการทดสอบซอฟต์แวร์**
   - เข้าใจหลักการและเทคนิคการทดสอบซอฟต์แวร์
   - สามารถออกแบบ Test Cases ที่มีคุณภาพและครอบคลุม
   - มีทักษะในการใช้เครื่องมือทดสอบอัตโนมัติ

2. **ประสบการณ์การทำงานจริง**
   - ได้รับประสบการณ์การทดสอบเว็บแอปพลิเคชันจริง
   - เข้าใจกระบวนการ Software Development Life Cycle (SDLC)
   - ฝึกฝนการแก้ปัญหาและการคิดวิเคราะห์

3. **ทักษะการจัดทำเอกสาร**
   - เรียนรู้การเขียนเอกสาร Test Plan และ Test Cases
   - พัฒนาทักษะการจัดทำรายงานการทดสอบ
   - ฝึกฝนการนำเสนอผลงาน

4. **การพัฒนาตนเอง**
   - เพิ่มความมั่นใจในการทำงาน
   - พัฒนาทักษะการทำงานเป็นทีม
   - เสริมสร้างทักษะการสื่อสาร

### 1.4.2 ประโยชน์ต่อระบบ

1. **การรับรองคุณภาพ**
   - ระบบมีคุณภาพและความน่าเชื่อถือสูงขึ้น
   - ลดความเสี่ยงในการเกิดข้อผิดพลาด
   - เพิ่มความมั่นใจในการใช้งานระบบ

2. **การค้นพบและแก้ไขข้อบกพร่อง**
   - ค้นพบข้อบกพร่องก่อนนำระบบไปใช้งานจริง
   - ลดต้นทุนในการแก้ไขข้อผิดพลาดในอนาคต
   - ปรับปรุงประสิทธิภาพการทำงานของระบบ

3. **เอกสารประกอบระบบ**
   - มีเอกสาร Test Cases สำหรับใช้ในการทดสอบต่อเนื่อง
   - มีแนวทางในการทดสอบสำหรับการพัฒนาเพิ่มเติม
   - มีรายงานคุณภาพของระบบที่เป็นปัจจุบัน

### 1.4.3 ประโยชน์ต่อองค์กร

1. **มาตรฐานการทดสอบ**
   - สร้างมาตรฐานการทดสอบที่เป็นแบบแผน
   - มีกระบวนการทดสอบที่สามารถนำไปใช้ซ้ำได้
   - เป็นต้นแบบสำหรับโครงการอื่นๆ

2. **ลดความเสี่ยง**
   - ลดความเสี่ยงในการเกิดปัญหาจากซอฟต์แวร์
   - เพิ่มความเชื่อมั่นของผู้ใช้งาน
   - ลดต้นทุนการบำรุงรักษาระบบ

3. **ความรู้และทักษะ**
   - ถ่ายทอดความรู้ด้านการทดสอบซอฟต์แวร์
   - เพิ่มศักยภาพของทีมพัฒนา
   - สร้างวัฒนธรรมการทดสอบในองค์กร

### 1.4.4 ประโยชน์ต่อสถาบันการศึกษา

1. **หลักสูตรการเรียนการสอน**
   - เป็นกรณีศึกษาสำหรับการเรียนการสอน
   - พัฒนาหลักสูตรให้ทันสมัยและตรงกับความต้องการของอุตสาหกรรม
   - เชื่อมโยงทฤษฎีกับการปฏิบัติจริง

2. **ความร่วมมือกับภาคอุตสาหกรรม**
   - เสริมสร้างความสัมพันธ์กับองค์กร
   - เปิดโอกาสให้นักศึกษาได้เรียนรู้จากผู้เชี่ยวชาญ
   - เพิ่มโอกาสในการจัดหางานให้นักศึกษา

## 1.5 แผนการดำเนินงาน

### 1.5.1 ระยะเวลาในการปฏิบัติงาน

**ระยะเวลา:** 12 สัปดาห์ (3 เดือน)  
**ช่วงเวลา:** [ระบุเดือน/ปี ตามจริง]

### 1.5.2 แผนการดำเนินงานแบ่งตามสัปดาห์

#### สัปดาห์ที่ 1-2: การเตรียมความพร้อมและศึกษาระบบ
- **สัปดาห์ที่ 1:**
  - ปฐมนิเทศและแนะนำองค์กร
  - ศึกษาโครงสร้างและฟังก์ชันของระบบ
  - ศึกษา Codebase และ Architecture
  - ทำความเข้าใจ Technology Stack

- **สัปดาห์ที่ 2:**
  - ศึกษาเอกสารและคู่มือการใช้งาน
  - ทดลองใช้งานระบบในบทบาทต่างๆ
  - จัดทำเอกสาร Test Plan
  - กำหนด Test Strategy

#### สัปดาห์ที่ 3-4: การออกแบบ Test Cases
- **สัปดาห์ที่ 3:**
  - วิเคราะห์ความต้องการในการทดสอบ
  - ออกแบบ Test Cases สำหรับ White Box Testing
  - ออกแบบ Test Cases สำหรับ Black Box Testing
  - สร้าง Test Data

- **สัปดาห์ที่ 4:**
  - รีวิวและปรับปรุง Test Cases
  - จัดทำเอกสาร Detailed Test Cases
  - เตรียม Test Environment
  - Setup เครื่องมือทดสอบ

#### สัปดาห์ที่ 5-7: พัฒนาการทดสอบอัตโนมัติ (White Box Testing)
- **สัปดาห์ที่ 5:**
  - พัฒนา Unit Tests สำหรับ Custom Hooks
  - พัฒนา Unit Tests สำหรับ Utility Functions
  - ทดสอบและแก้ไข Test Cases

- **สัปดาห์ที่ 6:**
  - พัฒนา Component Tests
  - ทดสอบการแสดงผลและ Interactions
  - วัด Code Coverage

- **สัปดาห์ที่ 7:**
  - พัฒนา Integration Tests
  - ทดสอบการทำงานร่วมกันของส่วนประกอบ
  - รันและวิเคราะห์ผลการทดสอบ

#### สัปดาห์ที่ 8-9: ดำเนินการทดสอบด้วยตนเอง (Black Box Testing)
- **สัปดาห์ที่ 8:**
  - ทดสอบฟังก์ชันการทำงานทั้งหมด (Functional Testing)
  - ทดสอบ User Interface
  - ทดสอบ User Flow และ Navigation

## 1.7 คำจำกัดความและศัพท์เฉพาะ

### คำศัพท์ด้านการทดสอบซอฟต์แวร์

**Software Testing** - กระบวนการตรวจสอบและประเมินคุณภาพของซอฟต์แวร์เพื่อให้มั่นใจว่าระบบทำงานตามที่คาดหวัง

**Test Case** - เอกสารที่ระบุขั้นตอนการทดสอบ ข้อมูลที่ใช้ทดสอบ และผลลัพธ์ที่คาดหวัง

**White Box Testing** - การทดสอบที่มองเห็นโครงสร้างภายในของโค้ด ทดสอบ Logic และ Code Path

**Black Box Testing** - การทดสอบที่ไม่มองเห็นโครงสร้างภายใน ทดสอบจากมุมมองของผู้ใช้งาน

**Unit Testing** - การทดสอบหน่วยย่อยที่สุดของโค้ด เช่น Function หรือ Method แยกส่วน

**Integration Testing** - การทดสอบการทำงานร่วมกันระหว่างส่วนประกอบต่างๆ ของระบบ

**System Testing** - การทดสอบระบบโดยรวมเพื่อตรวจสอบว่าตรงตามความต้องการทั้งหมด

**Automated Testing** - การทดสอบที่ใช้เครื่องมือหรือสคริปต์ในการทดสอบอัตโนมัติ

**Manual Testing** - การทดสอบที่ผู้ทดสอบดำเนินการด้วยตนเอง

**Test Coverage** - เปอร์เซ็นต์ของโค้ดที่ถูกทดสอบโดย Test Cases

**Bug/Defect** - ข้อบกพร่องหรือความผิดพลาดที่พบในระบบ

**Regression Testing** - การทดสอบซ้ำเพื่อตรวจสอบว่าการแก้ไขไม่ส่งผลกระทบต่อส่วนอื่นๆ

### คำศัพท์เฉพาะของระบบ

**Anime** - ภาพยนตร์แอนิเมชันหรือซีรีส์แอนิเมชันจากประเทศญี่ปุ่น

**Genre** - หมวดหมู่หรือประเภทของอนิเมะ เช่น Action, Romance, Comedy

**Back Office** - ส่วนของระบบสำหรับผู้ดูแลระบบในการจัดการข้อมูล

**Front Office** - ส่วนของระบบสำหรับผู้ใช้งานทั่วไปในการดูและค้นหาข้อมูล

**CRUD** - Create, Read, Update, Delete - การดำเนินการพื้นฐานกับข้อมูล

### คำศัพท์ทางเทคนิค

**React Component** - หน่วยการสร้าง UI ใน React ที่สามารถนำกลับมาใช้ใหม่ได้

**Hook** - ฟังก์ชันพิเศษใน React ที่ใช้จัดการ State และ Side Effects

**TypeScript** - ภาษา JavaScript ที่มีการกำหนดชนิดข้อมูล (Type)

**API (Application Programming Interface)** - ชุดคำสั่งสำหรับการสื่อสารระหว่างระบบ

**Mock** - การจำลองพฤติกรรมของส่วนประกอบเพื่อใช้ในการทดสอบ

**Snapshot** - การบันทึกสถานะของ Component เพื่อเปรียบเทียบในอนาคต

**CI/CD** - Continuous Integration/Continuous Deployment - กระบวนการอัตโนมัติในการ Build และ Deploy

**Repository** - พื้นที่จัดเก็บโค้ดและประวัติการเปลี่ยนแปลง

บทที่ 1 นี้ได้นำเสนอภาพรวมของโครงการสหกิจศึกษาในหัวข้อ "การออกแบบ Test Case และการวิเคราะห์ผลการทดสอบสำหรับเว็บแอปพลิเคชัน" โดยครอบคลุมความเป็นมา วัตถุประสงค์ ขอบเขต ประโยชน์ที่คาดว่าจะได้รับ แผนการดำเนินงาน และเครื่องมือที่ใช้ในโครงการ

โครงการนี้มุ่งเน้นการพัฒนาทักษะการทดสอบซอฟต์แวร์ทั้งแบบอัตโนมัติ (White Box Testing) และแบบด้วยตนเอง (Black Box Testing) บนระบบจัดการข้อมูลอนิเมะ โดยใช้เครื่องมือและเทคโนโลยีสมัยใหม่ เพื่อให้มั่นใจว่าระบบมีคุณภาพและความน่าเชื่อถือก่อนนำไปใช้งานจริง
